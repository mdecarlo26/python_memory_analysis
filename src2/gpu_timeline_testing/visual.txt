
========================
What the output means
========================

1) [CB][ENTER/EXIT][CPU] ...  (CUPTI Callback API)
   - These are CPU-side events emitted immediately when your program calls CUDA APIs.
   - Key fields:
     * t=... ns     : CPU timestamp (steady clock). Great for "what the app did when".
     * corr=ID      : correlation ID (a causal ID) for joining to GPU work.
     * cudaMemcpy / cudaMalloc / cudaLaunchKernel / cudaDeviceSynchronize :
                    : what the app requested from CUDA.
     * dur=... us   : time spent in that CPU API call itself (NOT kernel runtime).

   Why it matters:
   - Shows *intent* and CPU orchestration: allocations, copies, launches, syncs.
   - Lets you attribute actions to threads (if you add thread id) and to phases (NVTX).

2) [ACT][...][GPU] ...  (CUPTI Activity API)
   - These are GPU-side events showing what actually ran on the GPU.
   - Key fields:
     * start/end    : GPU timestamps (device clock domain).
     * dur=... us   : real GPU duration for the activity.
     * stream=...   : which CUDA stream executed it.
     * corr=ID      : join key back to the CPU API call that triggered this work.

   Why it matters:
   - This is *ground truth* runtime: kernel execution time, memcpy engine time, overlap.

3) [MEMINFO][CPU] ...  (cudaMemGetInfo sampling)
   - Snapshots of free/used device memory from the CPU point of view.
   - Key fields:
     * free/total/used MB : device memory pressure over time.

   Why it matters:
   - Lets you see allocation spikes, leaks, and whether you're approaching OOM.

========================
The critical idea: corr=...
========================
corr is NOT statistical correlation.
corr is a causal ID that lets you join:

  CPU:  cudaMemcpy (corr=6)   --> GPU: MEMCPY HtoD (corr=6)
  CPU:  cudaLaunchKernel (corr=8) --> GPU: KERNEL ... (corr=8)

That join is how you build one unified "CPU↔GPU memory dynamics timeline".
